# Sesion 2026-05-04 — Fase 5 CU16 CRM pacientes

## Objetivo
Implementar el modulo CRM de pacientes en backend respetando el patron de Fases 1-4, manteniendo prefijo `/api/` y sin tocar frontend.

## Implementado
- Modulo nuevo `backend/apps/crm/` con:
  - `SegmentacionPaciente`
  - `CampanaCRM`
  - `HistorialContacto`
- CRUD base DRF para los tres recursos con router:
  - `/api/crm-segmentaciones/`
  - `/api/crm-campanas/`
  - `/api/crm-contactos/`
- Permisos por rol:
  - mutaciones: `IsAuthenticated + IsAdministrativoOrAdmin`
  - lectura: `IsAuthenticated`
- Bitacora de create/update/delete para las tres entidades (`modulo='crm'`).
- Integracion en:
  - `backend/config/settings.py`
  - `backend/config/urls.py`
- Migracion inicial `0001_initial.py`.
- Tests minimos en `backend/apps/crm/tests/test_crm.py`:
  - permiso
  - validacion esencial de fechas
  - CRUD base y bitacora

## Evidencia de validacion
Comando ejecutado:
`python -m pytest apps/crm/tests/test_crm.py -q`

Resultado real:
- `ModuleNotFoundError: No module named 'rest_framework'`
- warning de pytest por config: `Unknown config option: DJANGO_SETTINGS_MODULE`

Conclusión: el entorno host de esta ejecucion no tiene dependencias backend completas (DRF / pytest-django).

## Riesgos / Notas
- Al habilitar lectura para todo usuario autenticado, perfiles clinicos no administrativos podrian consultar data CRM; si negocio exige confidencialidad estricta, cerrar lectura tambien a `IsAdministrativoOrAdmin`.
- Falta correr la suite en entorno Docker/venv oficial para validar migraciones y tests de punta a punta.
